# 第一部分：三个第一性原理 — 为什么必须这样做

在进入任何具体架构之前，需要先理解三个不可约的问题。这三个原理决定了后面所有架构选择不是「可以这样做」，而是「必须这样做」。

---

## 原理 1：知识必须复利，不能每次都重算

### 问题本质

RAG（检索增强生成）的工作方式是：每次用户提问，系统从原始文档库中检索相关片段，拼凑进 prompt，让模型现场合成答案。第 1 次和第 1000 次查询流程完全一样——检索、拼凑、合成。

这有三个根本缺陷：

**缺陷 1：无积累。** 模型每次都在「重新发现」知识。你上周问过的、模型已经深度分析过的东西，这周问同样的问题，它仍然要从原始文档重新检索、重新理解、重新推理。所有中间推理成果全部丢弃。

**缺陷 2：碎片化。** RAG 返回的是孤立的 chunk 片段，不是完整的知识结构。问一个需要综合 5 份文档才能回答的问题（比如「A 产品的定价策略在 Q3 调整后，对 B 渠道的利润率产生了什么影响」），RAG 返回 5 个不相关的片段，模型需要现场把这些碎片拼成逻辑链。拼错一个环节，整个答案就歪了。

**缺陷 3：矛盾盲区。** 你的知识库里有两份文档——1 月的定价文档说「B 渠道佣金 15%」，6 月的合同附件说「B 渠道佣金调整为 12%」。RAG 检索可能只返回其中一份，也可能两份都返回但模型没有注意到时间差。没有任何机制标记「这两条信息互相矛盾，需要人工确认」。

### Karpathy 的解法：LLM 是维基维护者，不是检索工具

Andrej Karpathy（OpenAI 联合创始人、前 Tesla AI 总监）在 2026 年初公开了一个个人知识管理的模式，核心思想一句话：

> 不要每次查询时从原始文档重新检索。让 LLM **增量构建并持续维护一个结构化的 wiki**——一个持久化的、互联的 markdown 文件集合，夹在你和原始资料之间。

具体做法：

```
传统 RAG：
原始文档 → 每次查询时检索 → 现场拼凑答案（无积累，每次一样贵）

Karpathy LLM Wiki：
原始文档 → LLM 编译一次 → 持久化 wiki（实体页+概念页+交叉引用+矛盾标记）
         → 查询时直接读这个已编译好的 wiki（复利效应，越用越聪明）
```

**关键区别**：wiki 是一个**持久化的、会自我增长的产物（compounding artifact）**。交叉引用已经建好了，矛盾已经标记了，综合结论已经反映了你读过的所有内容。每加一个新来源，wiki 变得更丰富、更互联。人类的工作从「维护知识库」变成了「策展来源 + 提好问题」。LLM 做所有脏活——总结、交叉引用、矛盾标记、索引更新。

### 这为什么是质的飞跃

对比两种方式的真实差异：

| | RAG | LLM Wiki |
|---|---|---|
| 第 1 次查询 | 检索 → 拼凑 → 回答 | 读 index → 定位相关页 → 回答 |
| 第 50 次查询 | 检索 → 拼凑 → 回答（和第 1 次一样贵） | 读 hot.md → 读 index → 定位页面 → 回答（更快，因为 wiki 已经结构化） |
| 新增第 51 个来源 | 索引一个新文档 | 提取实体/概念 → 创建新页面 → 更新所有相关页面 → 标记矛盾 → 更新索引。wiki 的整体价值上升 |
| 发现矛盾 | 模型可能注意到，也可能不会 | 已被标记为 `[!contradiction]` callout，查询时主动暴露 |
| 人类可读性 | 无法脱离 LLM 浏览知识库 | 在 Obsidian 中直接浏览，有 graph view、wikilinks、全文搜索 |

Karpathy 的原话：

> "The wiki is a persistent, compounding artifact. The cross-references are already there. The contradictions have already been flagged. The synthesis already reflects everything you've read. The wiki keeps getting richer with every source you add and every question you ask."

翻译：**wiki 是持续增长的复利资产。交叉引用已建好，矛盾已标记，综合结论已反映一切。每加一个来源、每问一个问题，wiki 就更值钱。**

### 工程含义：Context Engineering 比 Prompt Engineering 重要 10 倍

如果你接受「知识必须复利」这个原理，那么决定 Agent 质量的就不是「提示词写得多好」，而是**喂进模型的上下文对不对、及不及时、有没有权限、矛不矛盾**。

Karpathy 模式里有一个关键但容易被忽略的设计：**好答案可以写回 wiki**。你问了一个对比分析的问题，模型给了你一个高质量的对比表——这个对比表不应该消失在聊天历史里，而应该作为一个新页面写回 wiki。这样你的探索也会在知识库里复利增长，而不仅仅是摄入的来源在增长。

这也是为什么 claude-obsidian 在 `/save` 命令之外，还设计了 `/autoresearch`（自主研究循环）——它不只是回答问题，它把研究结果也写回 wiki。

## 原理 2：模型倾向做对 ≠ 一定能做对；约束必须外置

### 问题本质

Prompt 和 Skill 的本质是**影响模型的倾向**——让模型更可能做对。但它们不能**保证**模型做对。

一个简单的测试：你在 system prompt 里写「永远不要修改生产配置文件」。模型在 100 次对话中遵守了 99 次。第 100 次，用户说「帮我优化一下 nginx 配置」，模型看到了一个 `proxy_read_timeout` 的参数，判断「这个值太小了，在生产环境会导致超时」，于是改了。

模型做对了吗？从工程角度看，它改了。从安全角度看，它越权了。从「我有没有在 prompt 里写过」的角度看，它违反了。但模型不会因为 prompt 里有这句话就在架构层面被拦住——prompt 是一段文字，不是一道防火墙。

### 可执行性分级：什么能靠 Prompt，什么必须靠外部工具

这个问题可以按可执行性精确分级：

**A 类：Prompt 可以直接约束（模型能稳定做到的）**

这些规则影响的是「下一次输出」，模型在生成时就适用，不需要跨任务记忆或外部状态：

- 输出格式固定（比如统一用中文回复、按特定模板输出）
- 区分事实/推断/猜测（「我推测原因是 X」vs「原因是 X」）
- 没验证就说未验证（「我检查了代码」× →「我读了文件 X，但未运行测试验证」✓）
- 优先小范围取证、不要默认大改
- 修改后说明改了什么

**为什么这些能做到**：这些规则直接影响下一次 token 的生成，不需要模型「记住」任何东西，不需要外部状态判断。模型在生成「原因是 X」之前，可以通过 thinking 判断「我有没有证据支持 X 是原因」，没有就改成「我推测」。

**B 类：Prompt 只能部分约束（需要配合 AGENTS.md 等结构化指导）**

这些规则需要模型做任务级别的判断，在复杂上下文中容易失控：

- 只读取 3-5 个关键文件（而非全仓扫描）——模型可能判断「这个 bug 可能需要看 12 个文件」
- 识别是否过度工程化——这是一个主观判断，模型容易偏保守或偏激进
- 判断是否真的有增量——「我只改一行就行」vs「我重构了整个模块」
- 不引入额外复杂度——模型可能在修一个 bug 时引入一个新的抽象层

**为什么这些不稳定**：这些规则需要模型在「做多少」和「做到什么程度」之间做权衡，而权衡标准往往是隐性的、依赖具体场景的。同样的行为（比如引入一个新工具函数），在一个项目里是「必要的抽象」，在另一个项目里是「过度工程化」。

**C 类：Prompt 几乎无法约束（必须外部系统兜底）**

这些规则要求跨任务状态、持续学习、或对抗模型的根本行为模式：

- **不重复犯同类错误** — 模型没有「上次我在这里栽过」的记忆。每次对话是独立采样的。需要外部错误案例库 + 回归测试集。
- **自动沉淀经验、自进化** — 需要版本化的规则库 + eval pipeline + 人工确认入库，不是模型自己能做的。
- **控制 API 成本和上下文成本** — 模型天然倾向「越多信息越好」，需要 Langfuse/OpenTelemetry 做外部成本监控。
- **高风险动作不自动执行** — 需要审批门（Approval Gate），不是 prompt 里写一句「不要擅自发邮件」。
- **不污染长期记忆** — 需要记忆分层（M1 Hierarchical Retention），把临时任务细节和长期项目规则存在不同的作用域。

### 一句话判断框架

```
Prompt 负责「让模型想做对」
工具和策略负责「模型做错也过不去」
测试和日志负责「错了能发现、能复盘、能回归」
```

三层缺一层，就是不可靠系统。只有 Prompt = 模型错了没人知道。有 Prompt + 测试 = 模型错了能发现但挡不住。三层齐全 = 模型做对时 Prompt 引导，做错时工具拦截，拦截不了的测试和日志兜底复盘。

## 原理 3：Agent 系统的真正价值 = 降低下一次部署成本

### 问题本质

大多数团队衡量 Agent 项目成功用的是单点指标：这个 Agent 省了多少人工时间、提高了多少转化率。这些指标是对的，但它们只衡量了**这一次**的价值。

真正的判断标准是：**下一次部署 Agent 时，能不能比这一次更快、更稳、更便宜。**

原因是：第一个 Agent 从零搭建（找数据、接工具、设权限、建评估集、写 SOP），成本是固定的。如果第二个 Agent 仍然从零开始，说明你的「Agent 建设能力」没有增长——你只是在做服务交付，不是在积累平台能力。

但如果你把第一个 Agent 建设中产出的**模板、连接器、MCP Server、Eval Case、权限规则、数据模型、工作流组件、场景 SOP** 都沉淀成可复用资产，第二个 Agent 就可以站在第一个的肩膀上，建设成本骤降。

### FDE 模式的 5 个判断指标中，最关键的是第 5 个

FDE（Forward Deployed Engineer，现场部署工程师）机制有 5 个判断是否有效的指标：

1. 是否进入真实工作流？——用户是否真的每天用，而非演示时才用
2. 是否产生业务增量？——是否减少了人工时间、提高了转化、降低了错误
3. 是否有可验证数据？——是否有日志、评估集、人工修改率、失败样例
4. 是否能沉淀共性组件？——一次项目经验能否变成模板、工具、规则、MCP Server、Eval Case
5. **是否降低下一次部署成本？**——第二个 Agent 是否比第一个更快、更稳、更便宜

**第 5 条为什么最关键**：前 4 条衡量的是单次项目的成功，第 5 条衡量的是能力的累积。每次都重来 = 你在做外包。每次都能复用更多 = 你在建平台。这个指标直接决定了 3 年后的竞争力——你的第 20 个 Agent 是用 1 天部署还是 3 周部署。

### Context Engineering 在这个原理中的位置

回到原理 1 中的 Karpathy Wiki 模式——为什么 wiki 必须是持久化的、可复用的 markdown 文件集合？因为如果每次项目都重建知识库，你的知识资产就在衰减。RAG 的索引可以复用，但 RAG 没有「编译过的知识结构」——那是每次查询时现场拼凑的。

Wiki 的优势恰恰在于**编译一次，永久复用**。第一个 Agent 场景（比如邮件 Agent）建起来的 wiki（邮件模板、合作阶段判断逻辑、风险标记规则），第二个 Agent 场景（比如客服 Agent）可以直接引用里面的「产品信息」「常见问题」「公司政策」页面，不需要从零建。这就是知识复利在 Agent 系统中的具体体现。

# 第二部分：Karpathy LLM Wiki 模式 — 完整的知识架构

## 三层结构

Karpathy 的原始 gist 定义了一个三层架构。claude-obsidian 在此基础上做了完整的工程化实现。

### 第 1 层：Raw Sources（原始资料层）

存放所有原始文档——文章、论文、图片、数据文件、会议转录、合同扫描件。这一层的核心约束是**不可变**：LLM 从这里读取，但永远不修改。它是 wiki 所有内容的唯一事实来源。

目录约定：`.raw/`（claude-obsidian 中这个目录在 Obsidian 里默认隐藏，避免人类浏览时被原始文件干扰）。

**为什么必须不可变**：如果 LLM 可以修改原始来源，provenance（出处追踪）就断裂了。wiki 页面上的每一条事实都有一个 `source:` 字段指回 `.raw/` 中的原始文件。如果原始文件被改过，这个指针就变成了死引用。

### 第 2 层：The Wiki（LLM 维护的知识层）

这是 LLM 拥有、LLM 维护的 markdown 文件集合。人类阅读它，LLM 编写和维护它。包含以下几类页面：

| 页面类型 | 内容 | 示例 |
|---------|------|------|
| **Entity pages（实体页）** | 人物、公司、工具、产品 | `wiki/entities/Andrej Karpathy.md` |
| **Concept pages（概念页）** | 模式、框架、方法论 | `wiki/concepts/LLM Wiki Pattern.md` |
| **Source pages（来源页）** | 每个原始来源的摘要 | `wiki/sources/article-xxx.md` |
| **Synthesis pages（综合页）** | 跨来源的综合分析 | `wiki/concepts/compounding-knowledge.md` |
| **Session pages（会话页）** | 通过 `/save` 保存的对话 | `wiki/sessions/2026-06-23-email-strategy.md` |

claude-obsidian 在 v1.8 版本引入了 4 种组织方法模式（Methodology Modes），解决了一个关键问题：不同的知识管理哲学需要不同的文件组织方式：

| 模式 | 哲学 | 文件组织规则 |
|------|------|------------|
| **Generic**（默认） | 无意见，保持 v1.7 行为 | `wiki/sources/`, `wiki/entities/`, `wiki/concepts/`, `wiki/sessions/` |
| **LYT** | 笔记靠链接组织，文件夹不重要。MOC（内容地图）是导航原语 | `wiki/mocs/<topic>-moc.md` + `wiki/notes/<atomic-note>.md` |
| **PARA** | 按可行动性组织（Projects, Areas, Resources, Archives） | `wiki/projects/`, `wiki/areas/`, `wiki/resources/`, `wiki/archives/` |
| **Zettelkasten** | 原子笔记 + 唯一 ID + 密集双向链接，无文件夹 | `wiki/<YYYYMMDDHHMMSSffffff>-<slug>.md`（平铺，时间戳命名） |

**为什么需要方法模式**：Karpathy 原始 gist 没有规定怎么组织文件。但实际使用中，500 页平铺在一个目录里会不可维护。不同的场景适合不同的组织方式——研究型项目适合 Zettelkasten 的原子笔记，业务型项目适合 PARA 的可行动性分类。claude-obsidian 把这个选择做成了可配置的选项，且切换模式不会自动迁移已有文件（避免破坏性操作）。

### 第 3 层：Schema（模式约定层）

这是一份文档（CLAUDE.md 或 AGENTS.md），告诉 LLM wiki 怎么结构化、命名规则是什么、摄入来源时遵循什么工作流、回答问题时怎么引用页面。这是整个系统最关键但最容易被忽略的一层。

**为什么 Schema 是独立的层而不是嵌在 prompt 里**：因为 Schema 需要人类和 LLM 共同演化。你在使用过程中发现「实体页应该加一个 `related_projects:` 字段」，就在 CLAUDE.md 里加上这条约定，下一次 LLM 摄入新来源时就自动遵守。如果把约定散落在每次对话的 prompt 里，你永远积累不起一套稳定的 wiki 维护规则。

## 三个核心操作

Karpathy 模式定义了三个操作。claude-obsidian 把它们工程化成了具体的 slash 命令：

### 操作 1：Ingest（摄入）— 命令：`ingest [source]`

**流程**：LLM 读取 `.raw/` 中的新来源 → 和你讨论关键要点 → 创建来源摘要页 → 提取实体和概念，创建或更新对应页面 → 在所有相关页面之间建立 wikilinks → 检查是否有内容与已有 wiki 页面矛盾，有则标记 `[!contradiction]` → 更新 `index.md`（目录）→ 在 `log.md` 追加一条操作记录。

**关键数据**：一次摄入通常触及 10-15 个 wiki 页面。claude-obsidian 的实测数据：从 30+ 个来源（研究论文、会议转录、技术文档）中，wiki-ingest agent 平均每个来源产出 8-15 个 wiki 页面，每个页面平均 12 个 wikilinks。25 个来源建的 200 页 vault 中，94% 的页面与至少 2 个其他页面相连——零手动链接。

**为什么每次摄入要触达这么多页面**：因为知识不是孤岛。你加一篇关于「Gmail API OAuth 配置」的文章，LLM 不仅要创建一个来源页，还要：
- 更新 `Gmail API` 实体页（如果有新的 API 端点发现）
- 更新 `OAuth 2.0` 概念页（如果有新的认证流程细节）
- 检查是否与已有的 `邮件发送 Agent` 页面中的实现步骤矛盾
- 如果文章提到了某个具体的错误码，更新 `错误案例` 页面

如果不做这些，wiki 就是一堆孤立摘要的集合，不是互联的知识图谱。

### 操作 2：Query（查询）— 命令：`what do you know about X?`

**流程**：先读 `wiki/hot.md`（热缓存，约 500 words，最近一次会话的上下文摘要）→ 读 `wiki/index.md`（完整页面目录，每个页面一行 + 一句话描述）→ 定位 3-5 个最相关的页面 → 深入读取这些页面 → 合成答案 + 引用具体的 wiki 页面（用 `[[wikilinks]]` 格式）。

**Token 预算控制**：hot.md ~500 tokens + index.md ~1000 tokens + 3-5 个页面 ~3000 tokens = 总计约 5000 tokens。即使 vault 有 300+ 页面，单次查询的 token 消耗仍然可控。这是 Karpathy 模式比 RAG 在成本上的核心优势之一——RAG 每次查询都要从向量库检索、重排、注入多个 chunk，token 消耗往往更高且不可预测。

**为什么答案可以写回 wiki**：Karpathy 原话——"good answers can be filed back into the wiki as new pages." 你问的一个对比分析、一个深度解读、一个你发现的关联——这些是有价值的产出，不应该消失在聊天历史里。claude-obsidian 的 `/save` 命令专门做这件事：把当前对话归档为一个结构化的 wiki 笔记。

### 操作 3：Lint（健康检查）— 命令：`lint the wiki`

**流程**：LLM 扫描整个 wiki，检查 8 类问题：

| 问题类型 | 说明 |
|---------|------|
| Orphan pages（孤立页） | 没有任何页面链接到它 |
| Dead wikilinks（死链） | 指向不存在页面的链接 |
| Contradictions（矛盾） | 不同页面之间的冲突声明 |
| Missing pages（缺页） | 被引用但从未创建的页面 |
| Unlinked mentions（未链接提及） | 文本中出现实体名但没有 wikilink |
| Incomplete metadata（元数据不全） | 缺少 frontmatter 字段 |
| Empty sections（空章节） | 有标题但没有内容的占位 |
| Stale index（过期索引） | index.md 与实际 vault 内容不一致 |

**为什么 Lint 是必要操作**：人类放弃 wiki 的根本原因不是「不会写」，而是「维护负担增长得比价值快」。交叉引用要手动更新、矛盾要手动发现、索引要手动维护——当 wiki 超过 50 页，维护成本就超过了人类愿意投入的时间。LLM 做 Lint 的优势恰恰在于它不会觉得枯燥、不会忘记更新交叉引用、能一次扫描 200 页发现所有矛盾。维护成本趋近于零，wiki 就不会变成数字墓地。

## claude-obsidian：Karpathy 模式的工程化实现

你已经在用 Karpathy 原始 gist 的 Obsidian 数据库。claude-obsidian 是 Agrici Daniel 在此基础上做的完整工程化实现（358+ GitHub stars、MIT 开源），核心创新包括：

### 1. Hot Cache（热缓存）— 消除「每次新对话重新解释一遍」的问题

**问题**：每次开启新对话，AI 对你的 vault、上次聊到哪里、做了哪些决策一无所知。你需要花 5-10 分钟重新解释上下文（约 2000-3000 tokens）。

**解法**：claude-obsidian 维护一个 `wiki/hot.md` 文件（约 500 words），包含：
- 上次会话你在做什么
- 做了哪些关键决策及原因
- 最近创建或更新了哪些 wiki 页面
- 未解决的问题或你提到的下一步

每次新会话启动时，LLM 先读 hot.md（约 500 tokens，不到 Claude 上下文的 0.25%），立即恢复工作记忆。投入 500 tokens，节省 2000-3000 tokens 的重复解释——4-6 倍的 ROI。

### 2. Hybrid Retrieval（混合检索，v1.7+）— 基于 Anthropic 2024 年 9 月研究

**问题**：纯语义检索（embedding → vector search）对精确关键词（产品名、错误码、特定 ID）经常 miss。纯关键词检索（BM25）对同义词和隐含语义关系又抓不住。

**解法**：三层检索管道：
1. **BM25 稀疏检索**：永远是开启状态，覆盖精确关键词匹配
2. **Contextual Prefix**（可选，需用户同意 API egress）：把页面正文发给 Anthropic API 生成上下文前缀，再 embed。Anthropic 2024 年 9 月的研究显示这个方法显著降低检索失败率
3. **Cosine Rerank**：用本地 ollama 模型对候选结果做重排

**实测效果**：在 50 条查询的 benchmark 上，相比 v1.6 基线：
- Top-1 准确率 +32 个百分点
- 错误率降低 41%
- 管道优雅降级：任何一层不可用时，其余层仍返回有效结果

### 3. Multi-Writer Safety（多写入者安全，v1.7+）

**问题**：并行 ingest 多个来源时，两个 sub-agent 可能同时尝试写入同一个 wiki 页面（比如同时更新同一个实体页），产生竞态条件导致文件损坏。

**解法**：per-file advisory locking（`scripts/wiki-lock.sh`）。一个 writer 获取锁后写入，另一个 writer 检测到锁就跳过并在下一轮重试。PostToolUse auto-commit hook 在提交前检查锁列表，延迟提交直到所有写入完成。过期锁 60 秒自动回收。这是把数据库事务的思维带进文件系统的 Agent 协作。

### 4. Autoresearch（自主研究循环）

**流程**：`/autoresearch [topic]` 触发 3 轮研究循环：

- **Round 1（广泛搜索）**：把主题拆成 3-5 个角度，每个角度跑 2-3 个搜索查询，取每个查询的 top 2-3 结果
- **Round 2（补缺）**：针对 Round 1 发现的矛盾和信息缺口做定向搜索
- **Round 3（综合检查，可选）**：如果前两轮后仍有重大缺口，再跑一轮
- **归档**：综合页 + 来源页 + 实体页 + 概念页，全部交叉引用

你可以在 `skills/autoresearch/references/program.md` 中自定义：偏好什么来源（学术、官方文档、新闻）、置信度评分规则、最大轮数和最大页面数。这不是盲目的爬虫，是带出处追踪的导向研究。

### 5. 15 个 Claude Code Skills 覆盖完整生命周期

从来源摄入到 vault 维护到可视化到深度思考，15 个 skill 形成闭环：

# 第三部分：ADPS 33 模式 — 统一的 Agent 设计语言

## 为什么需要一套设计模式语言

市面上大多数「Agent 架构」指南给的是平铺清单——Reflection、ReAct、Multi-Agent、Tree of Thoughts……平铺清单回答的是「有哪些模式」，但它不回答：

- **我的问题落在哪个坐标上？那个坐标上住着哪个模式？**
- **一个贷款评估 Agent 崩掉，是因为缺了 Reflection，还是因为感知阶段的上下文预算分配把关键文档丢了？**
- **一个多 Agent 代码评审系统漂移，是因为 Agent 不够强，还是因为协作拓扑缺少冲突解决机制？**

**ADPS 用双轴矩阵解决了这个问题**：知道失败发生在哪条轴上，比知道失败模式叫什么名字更重要。

## 双轴结构

### 行：7 个认知功能（Agent 在做什么类型的工作）

| 功能 | 核心问题 | 你的 Hermes 覆盖 |
|------|---------|-----------------|
| **感知 Perception** | 信息怎么进 context？怎么分优先级？太多了怎么压？未知空间怎么探？ | P1-P4 全部覆盖 |
| **记忆 Memory** | 什么信息跨 session 保存？知识库怎么检索？做到哪了？踩过什么坑？ | M1-M5 全部覆盖 + MxR 记忆路由器 |
| **推理 Reasoning** | 怎么得出结论？简单问题用便宜模型？复杂问题怎么拆分？ | R1-R5 全部覆盖 |
| **行动 Action** | 选哪个工具？任务怎么拆？执行顺序？工具太多怎么办？错了谁兜底？ | A1-A5 全部覆盖 + AxP 并行 + AxL 重试 |
| **反思 Reflection** | 输出好不好？怎么改进？成功经验怎么沉淀？失败怎么自动修？ | F1-F4 全部覆盖 |
| **协作 Collaboration** | 多 Agent 怎么分工？各干各的还是互相审查？ | C1-C6 全部覆盖 + CxR 路由 + CxO 编排 |
| **治理 Governance** | 什么能做？做错了最多毁多少？权限怎么渐进给？怎么观测？ | G1-G5 全部覆盖 + GxC 治理检查链 |

### 列：6 个执行拓扑（工作长什么形状）

| 拓扑 | 速记 | 适合什么 |
|------|------|---------|
| **链式 Chain** | 传 | 步骤稳定，前一步输出直接给后一步。比如：校对 → 改写 → 核查数字 → 起标题 |
| **路由 Route** | 选 | 先分类，再选模型/工具/流程。比如：简单问题用 cheap model，复杂问题用 expensive model |
| **并行 Parallel** | 撒 | 多路同时展开，用成本换质量或时间。比如：3 个 Agent 同时审同一份合同，取最保守结论 |
| **编排 Orchestrate** | 协 | 一个中心协调者维护全局目标、任务账本、最终汇总。比如：supervisor 拆任务→派给 worker→合并产出 |
| **循环 Loop** | 转 | 生成、观察、修正、再生成，直到收敛。比如：假设→验证→证伪→新假设 |
| **层级 Hierarchy** | 分 | 多层职责、多层权限、多层记忆、多层防护边界。比如：L1 只读→L2 打草稿→L3 自动执行可逆动作 |

7 行 × 6 列 = 42 格。ADPS 定义了 28 个核心模式（落在已命名的格子里），5 个扩展模式（M5 程序性记忆、R5 双模架构、A4 最简工具集、C5 子代理隔离、G4 钩子流水线），加上新兴拓扑 C6 编舞，合计 34 篇模式白皮书。

**14 个空格是有意空白**——ADPS 的立场是「不要为了形式对称去发明不成熟的模式」。空格代表工业空白和未来研究前沿。

### 你的本地覆盖状态

你已验证的 ADPS 本地安装覆盖了 34/42（81%），包括全部 28 核心 + 全部 6 扩展模式 + 6 个新增空白格覆盖（MxR 记忆路由器、AxP 并行行动、AxL 重试循环、CxR Agent 路由器、CxO Agent 编排器、GxC 治理检查链）。

剩余 8 个基础设施缺口（记录在 `adps-infra-gaps` skill 中）需要 Hermes 平台层的架构级支持才能实现，不是当前阶段可以靠创建新 skill 解决的。这些是路线图上的未来事项，不是阻碍。

## 五个工程契约 — 每个 Agent 必须显式设计的 5 个接口

ADPS 在矩阵之外还定义了 5 个工程契约。它们是 Agent 从「能跑通的 demo」变成「可控的生产系统」的关键。每一个契约回答一个不可回避的问题：

### 1. Context Contract（上下文契约）

**回答**：什么信息进上下文？什么延迟检索（只挂句柄，需要时再拉）？什么直接丢弃？

**为什么必须有**：Agent 最常见的故障模式不是信息不够，而是信息太多。一个中等 codebase 加三周 git 历史扔进 context 就是 180K token 起步。如果没有明确的上下文预算分配，关键信息会被过期日志、无关文件、历史对话淹没。模型看着「完整一套」给出错误结论。

**对应 ADPS 模式**：P1 上下文分诊（感知×路由）负责这件事。

### 2. Evidence Contract（证据契约）

**回答**：一条证据必须带 source（出自哪里）、version（什么版本）、scope（适用范围）、citation（原文引用）、provenance（获取路径）。RAG 管业务证据（文档里的知识），SessionState 管机械真值（employee_id、amount、approval_id 这类不能靠模型合成的东西）。

**为什么必须有**：没有证据契约的 Agent 会「自信地说出无法追溯的内容」。监管来查时拿不出依据，出了事无法 root-cause，团队无法判断一个结论是模型从知识库里正确引用的还是幻觉。

**对应 ADPS 模式**：M2 RAG（记忆×链式）管业务证据检索，M1 分层保留（记忆×层级）管不同作用域的证据生命周期。

### 3. Decision Contract（决策契约）

**回答**：模型的输出怎样变成结构化的 RouteDecision / PlanDecision，而不是一段自然语言？

**为什么必须有**：自然语言决策不可被程序化校验。模型说「我觉得应该先查订单状态再判断是否退款」——这是一段话，不是一个可以被下游工具链解析、验证、记录的结构。结构化决策（JSON schema）才能被 policy engine 拦截（「金额超过 500 的退款决策必须人工审批」）、被日志记录、被评估集回归测试。

**对应 ADPS 模式**：R1 思维链（推理×链式）管推理过程的结构化存储，R2 复杂度路由（推理×路由）管决策分发到合适的模型档位。

### 4. Action Contract（行动契约）

**回答**：每个工具调用的元数据（是不是只读？能不能并发？是不是破坏性的？需不需要刷新状态？需不需要审批？参数从哪里来？失败了怎么回滚？）

**为什么必须有**：工具是 Agent 所有副作用的来源。没有行动契约，一个被 prompt injection 劫持的 Agent 可以发送邮件、删除文件、修改数据库——每一步都「合法」因为工具接口允许。行动契约把「能不能调」从模型判断权中拿走，交到工程层的元数据声明里。

**对应 ADPS 模式**：A1 工具调度（行动×路由）管选哪个工具，A5 守卫三明治（行动×层级）在工具调用前后套 pre/post check，G1 审批门（治理×路由）管高风险动作的人工审批。

### 5. Trace Contract（追踪契约）

**回答**：每个 LLM call、tool call、handoff、approval、state update 是否串到同一个 trace id？是否可以按 trace_id 回放完整调用链？

**为什么必须有**：Agent 的故障方式跟传统软件不同——传统软件在某个时间戳抛异常，stack trace 一目了然。Agent 是慢慢漂移的：完全正确 → 仍然正确 → 看起来正确 → 自信但错 → 沉默灾难化，没有一个 crash point。没有完整的 trace，漂移过程中哪个环节开始出问题完全不可查。

**对应 ADPS 模式**：G5 可观测性（治理×编排）管这件事。

## 三套起步组合

不需要把所有 33 个模式都装进系统。根据你的业务类型，选对应的组合即可。注意：任何带写权限的 Agent，G1 审批门和 G5 可观测性是底线——不是可选的。

### 组合 1：知识问答型 Agent

**你的对应场景**：电商客服 RAG、广告 SOP 知识库检索

**模式组合**：P1 上下文分诊 + M2 RAG + R2 复杂度路由 + G5 可观测性

**边界提醒**：RAG 只提供业务证据，不提供机械真值（订单金额、客户 ID 这类），所有引用必须可追溯到来源页面。

### 组合 2：执行型 SaaS Agent

**你的对应场景**：邮件回复 Agent、商品选品 Agent

**模式组合**：P1 + M2 + M3 进度追踪 + R2 + A1 工具调度 + A2 规划执行 + G1 审批门 + G2 爆炸半径控制 + G5

**边界提醒**：employee_id、amount、approval_id 这类机械状态必须来自工具调用和 SessionState，不能让模型当场合成。审批门必须对「正式发送邮件」这类动作做硬拦截。

### 组合 3：多 Agent 研究或审查系统

**你的对应场景**：视频拆解与方法论沉淀（多 Agent 各拆一个维度）

**模式组合**：P3 渐进发现 + M2 + R3 并行探索 + C1 层级委派 + C2 扇出聚合 + C3 对抗评审 + G5

**边界提醒**：多 Agent 的收益来自任务规模、角色边界清晰、独立审查——不来自「看起来更先进」。3 个 Agent 以下的场景单 Agent 通常更稳。

## 六步选型法：从业务问题到模式组合

ADPS 定义了一个 15 分钟可完成的选型流程：

1. **界定（Bound）**：写一页范围声明。这个 Agent 做什么？不做什么？边界画在哪？
2. **映射（Map）**：对 7 个认知功能逐行打分——None（不需要）/ Light（轻度需要）/ Heavy（重度需要）
3. **选拓扑（Select）**：按任务的主约束确定执行拓扑。步骤确定 → 链式。需要分类 → 路由。需要并行提速 → 并行。需要中心协调 → 编排。需要试错收敛 → 循环。需要多层隔离 → 层级。
4. **挑模式（Pick）**：在矩阵上按坐标挑 5-7 个模式。不是 28 个全装。
5. **校预算（Assess）**：用准确率、延迟、成本三轴校验这个组合是否合理。
6. **分步落地（Build）**：先跑通最小闭环（核心模式），再叠反思记忆（经验回放、技能包），最后叠治理（审批门、可观测性）。

## Context Engineering 在 ADPS 中的精确位置

回到你之前问的问题：Context Engineering 和 ADPS 是什么关系？

**Context Engineering ⊂ ADPS**。Context Engineering 是 ADPS 感知行（P1-P4） + 记忆行的检索部分（M2）。ADPS 比 Context Engineering 多出来的是推理、行动、反思、协作、治理这 5 个完整的认知功能维度。

具体对应：

| Context Engineering 要做的事 | ADPS 模式 | 为什么需要这个模式 |
|---|---|---|
| 上下文太多，需要优先级分层 | **P1 上下文分诊** | 把候选信息分 P0/P1/P2/P3 四级，按 token 预算从高到低塞。保证最关键信息不被淹没 |
| context 满了，需要压缩 | **P2 语义压缩** | 三层级联压缩（截断→摘要→深压），但要保护错误堆栈和关键数字不被压掉 |
| 面对未知信息空间，需要探索 | **P3 渐进发现** | 广扫→精读→深追三阶段。不把整个 codebase 喂进去，而是逐步定位相关信息 |
| 输入混合了图/表/文/日志 | **P4 多模态融合** | 图转 Mermaid（省 24× token），表格转 markdown（省 9×），chart 保留图但数字走 CSV 提取 |
| 知识库检索 | **M2 RAG** | Naive RAG（一次查询一次返回）+ Agentic RAG（多步检索、迭代改写 query、跨语料交叉验证） |
| 跨 session 记忆 | **M1 分层保留** | 按作用域分层存储（用户层/项目层/会话层/临时层），每层独立 TTL 和加载策略 |

Context Engineering 的 6 个能力在 ADPS 里都有精确的模式坐标和执行拓扑。它的价值在于「决定 Agent 看到了什么」。ADPS 的另外 5 行决定了「看到之后怎么想、怎么做、怎么改、怎么配合、怎么被管住」。

# 第四部分：8 层工程治理栈 — 从 Demo 到生产系统的完整约束链

## 为什么需要治理栈

第三部分讲了怎么设计 Agent（用 ADPS 模式矩阵），这一部分讲怎么管住 Agent。这两件事是独立的——设计得好不等于管得住。

核心判断：

> Prompt 和 Skill 只能让模型「倾向于做对」。工程治理栈让模型「做错也过不去」。
> 测试和日志让模型「错了能发现、能复盘、能回归」。

**8 层治理栈中的每一层都回答一个「过不去」的问题**。如果某一层缺失，那个维度上的风险就没有兜底机制。

---

## 第 1 层：工具接入层 — 模型怎么调外部系统？

**工具**：MCP / OpenAPI / FastMCP / PydanticAI Tools / Function Calling

**解决的问题**：把模型能调用的外部能力封装成明确的工具接口，而不是让模型自由发挥。MCP 可以统一接入 GitHub、Gmail、Shopify、数据库、文件系统、广告数据、知识库。

**关键边界**：MCP 只负责「怎么接工具」。不负责这三个致命问题：
- 该不该允许调用？
- 调用后是否安全？
- 结果是否正确？

这三个问题由后面的层解决。把 MCP 当治理系统是危险的——它只是接口层。

---

## 第 2 层：权限与策略层 — 什么动作不允许？

**工具**：OPA（Open Policy Agent）/ Conftest / Policy Gateway / Rego 规则 / RBAC / ABAC

**解决的问题**：把「禁止事项」从 prompt 里的文字变成机器可执行的规则。

可定义的典型规则：
- 没有人工审批 → 不允许发送邮件
- 没有测试通过 → 不允许修改生产配置
- 涉及金额/退款/合同/报价 → 必须人工确认
- 高风险文件 → 只能读，不能写
- 没有订单权限 → 不允许查询用户隐私数据

**为什么 prompt 里写「不要越权」不够**：因为 prompt 需要被模型「理解」才能生效，而权限规则需要被系统「强制执行」。这两件事之间有本质鸿沟——模型可以被 prompt injection 绕过，可以被复杂任务分散注意力，可以在多轮对话中「忘记」这个约束。OPA 的策略规则是代码，不可能被绕过。

**执行路径**：
```
模型请求工具调用 → Policy Gateway 检查 → 通过（放行）/ 拒绝（返回不可覆盖的错误）
```

---

## 第 3 层：执行隔离层 — Agent 最多能破坏什么？

**工具**：Docker / Dev Containers / Git worktree / 只读挂载 / 沙箱 / 文件白名单 / 命令白名单 / 网络隔离 / 密钥隔离

**解决的问题**：不指望 Agent 不失败，只确保失败时损失被锁死在可控范围内。

典型规则：
- 代码 Agent 默认在临时 Git worktree 或分支执行（不改主分支）
- 敏感目录只读挂载（`.env` 不暴露给 Agent）
- 生产数据库不可直连（Agent 只能访问 staging）
- 外部网络默认关闭（防止数据外泄）
- 危险命令禁止：`rm -rf`、`curl | bash`、`deploy`、`terraform apply`

**为什么隔离是治理而非运维**：隔离的目的不是「提供一个运行环境」，而是「定义最大可接受损失」。如果一个 Agent 被 prompt injection 完全劫持，它在一个只读挂载 + 网络关闭 + 命令白名单的沙箱里能造成的最大损失是什么？这个答案应该在部署前就被量化，而不是出事后才发现。

---

## 第 4 层：工作流状态层 — 长任务断了能恢复吗？

**工具**：Temporal / n8n / Prefect / Dagster / Airflow / Hermes cron / GitHub Actions

**解决的问题**：Agent 的动作变成可恢复、可重试、可审计的流程，而不是一次性的聊天输出。如果流程中途崩溃（Agent 超时、API 限流、外部系统不可用），能从断点恢复而不是重头来。

典型流程模式：
```
事件触发 → Agent 判断 → 查数据 → 调工具 → 生成草稿 → 等人工确认 → 执行动作 → 写日志 → 后续跟进
```

**n8n vs Temporal 的选择**：n8n 适合低代码 workflow（你能在可视化界面上拖拽节点编排流程），Temporal 适合工程级长流程和高可靠任务（支持 Saga 事务补偿、长期运行的工作流状态持久化、多语言的 SDK）。

**Hermes 覆盖**：Hermes 的 `cronjob` + `context_from`（链式 cron：job A 产出 → 注入 job B 的 prompt）可以实现简化版的 workflow 编排。对于邮件 Agent 场景（定时拉取邮件 → 判断 → 生成草稿 → 人工审批 → 发送），这个组合已经够用。

---

## 第 5 层：验证门禁层 — 「已验证」是真跑了还是口头说的？

**工具**：pytest / CI / Semgrep / CodeQL / pre-commit / ESLint / Ruff / mypy / Playwright

**解决的问题**：让「已验证」变成真实命令的返回结果，而不是模型的口头保证。模型说「测试通过」不算通过，CI 绿了才算。

工程规则：
- 没有测试结果 → 不允许声明「完成」
- CI 不通过 → 不允许合并
- 高风险改动 → 必须人工 code review
- 静态扫描失败 → 不允许进入主分支

**为什么这一层对 Agent 尤为重要**：Agent 的一个核心故障模式是「自信地说测试通过了但其实没跑」。这不是恶意欺骗，而是 LLM 的生成行为——在它被训练的数据中，「我检查了代码，看起来没问题」是一个高频出现的模式。模型不是在有意识地撒谎，是统计学上这句话在类似上下文中经常出现。

---

## 第 6 层：LLM 观测层 — 哪一步错了？花了多少 token？

**工具**：Langfuse / OpenTelemetry / Arize Phoenix / Helicone / PromptLayer / 自建 Postgres 日志表

**解决的问题**：记录 Agent 每一步为什么这么做。企业不能接受黑盒 Agent。

必须知道的：
- 用了哪个 Prompt 版本？
- 检索了哪些资料？（哪些没检索到？）
- 调用了哪些工具？哪个失败了？
- 哪一步开始出错的？
- 一次任务花了多少 token 和 API 成本？
- 人工修改了多少？哪个场景修改率最高？

完整链路：`输入 → 检索 → 推理 → 工具调用 → 输出 → 人工修改 → 最终结果 → 业务指标`

**没有这一层，就没有持续优化**。你不知道改了一个 prompt 是好是坏，不知道哪个场景的 Agent 成本失控，不知道哪个工具调用失败率在悄悄上升。

---

## 第 7 层：评估层 — 系统真变好了还是看起来变好了？

**工具**：Ragas / DeepEval / promptfoo / OpenAI Evals / Golden Set（黄金测试集）

**解决的问题**：用测试集判断 Agent 质量是否真的在提升，而不是靠感觉。每次改 Prompt、知识库、工具、模型，都跑一次完整的评估。

Golden Set 规模建议：
| Agent 类型 | Golden Set 规模 | 为什么这个规模 |
|-----------|-----------------|--------------|
| 客服 Agent | 50-100 条真实客户问题 | 覆盖高频问题和边界 case |
| 邮件 Agent | 50-100 条历史邮件 | 覆盖不同合作阶段和风险等级 |
| 广告 SOP Agent | 30-50 个投放错误案例 | 每类错误至少 3 个变体 |
| 视频拆解 Agent | 20-30 个标准拆解样例 | 覆盖不同视频类型 |

**为什么评估层是独立的一层而非测试的一部分**：传统软件的测试判断标准是二元的——通过/失败。Agent 的输出质量是连续的——「还可以，但漏了一个风险点」「方向对，但引用错了来源」。你需要评估框架（如 Ragas 的 faithfulness、answer relevancy、context precision）给出多维度的分数，而不仅仅是 pass/fail。

---

## 第 8 层：输出与结构校验层 — 输出格式对不对？有没有风险内容？

**工具**：Pydantic / JSON Schema / Guardrails AI / NVIDIA NeMo Guardrails / PydanticAI

**解决的问题**：限制模型输出的格式、字段、风险内容。结构化的输出才能被下游工具链解析、验证、记录。

示例结构化输出（邮件 Agent）：
```json
{
  "intent": "collaboration_reply",
  "risk_level": "medium",
  "needs_human_review": true,
  "reason": "creator mentioned paid rate",
  "draft_reply": "...",
  "next_action": "manual_price_review"
}
```

**为什么结构校验重要**：如果模型的输出是自由文本，「需要人工审核」可能是文本中的一句话但不保证被下游系统解析到。结构化字段 `"needs_human_review": true` 可以被 Policy Gateway 直接读取——「如果 needs_human_review=true，阻断自动发送，路由到人工审核队列」。

---

## MCP 的正确位置：接口 vs 治理

基于以上 8 层，MCP 的正确架构不是：

```
❌ LLM → MCP Server → 直接操作 Gmail/GitHub/数据库
```

而是：

```
✅ LLM → MCP Client → Policy Gateway（权限+审计）→ MCP Server → 实际工具/数据源 → 日志/审计/验证
```

MCP 的权限应该分为三个等级：

| 等级 | 允许的操作 | 要求 |
|------|----------|------|
| **Read-only** | 读代码、查文档、查历史邮件、查订单状态、查广告数据 | 默认开放 |
| **Draft-only** | 创建邮件草稿、生成 PR、生成配置变更、写入待审核表 | 需要日志记录 |
| **Action** | 发送邮件、修改数据库、部署代码、更新 Shopify、删除文件 | **必须人工审批** |

---

## 三级工程治理栈：按组织规模选择

| 级别 | 适用场景 | 工具链 |
|------|---------|--------|
| **Tier 1** | 个人开发者/小团队 | AGENTS.md + Git worktree + pytest/lint + pre-commit + Semgrep + MCP read-only + 人工确认 |
| **Tier 2** | 企业内部 Agent | Tier 1 + Pydantic/JSON Schema + Policy Gateway + n8n/Temporal + Langfuse + Ragas/Golden Set + 人工审核表 |
| **Tier 3** | 生产级平台 | Tier 2 + OPA + RBAC/ABAC + OpenTelemetry + CI/CD + Secrets Manager + Sandbox Runtime + Eval Pipeline + Audit Log |

Tier 1 典型配置包括 AGENTS.md、MCP 的 read-only 配置（Gmail OAuth 只读）、以及通过本地模型代理控制模型调用。下一步是补验证门禁（为邮件 Agent 建 Golden Set）和补观测层（token 消耗和错误追踪）。

# 第五部分：FDE — 从 Demo 到生产系统的交付机制

## FDE 是什么：不是岗位，是组织机制

FDE（Forward Deployed Engineer）这个词容易被理解成一个新岗位——「派工程师去客户现场写代码」。这个理解没错，但太浅。

FDE 的真正内涵是三层：

### 第 1 层：系统交付层（核心）

**FDE 做的事**：把抽象的 LLM 能力翻译成真实企业可以运行的系统。

翻译的内容：
- **企业数据在哪** → 接进 Agent 的上下文
- **企业工具有哪些** → 封装成 MCP Server / Function Calling
- **企业流程怎么走** → 编码成 Workflow（哪步自动、哪步人工审批）
- **企业权限怎么分** → 映射成 Policy 规则
- **企业怎么判断好坏** → 建立评估指标和 Golden Set
- **出事了谁负责** → 定义责任边界（人审批的 = 审批人负责，自动执行的 = 产品 owner 负责）

**为什么需要 FDE 而不是产品经理 + 后端工程师的组合**：因为企业 Agent 的难点不是「模型会不会回答」，而是模型的回答需要接入企业的真实数据、真实权限、真实流程。这三件事是交织在一起的，不可能在没有业务现场理解的情况下单独完成。

### 第 2 层：产品反哺层（决定 3 年后的竞争力）

**FDE 不应该每个客户做一套独立系统**。正确的路径是：

```
先为一个场景定制 → 验证真实价值 → 发现共性问题 → 抽象成可复用模块 → 反哺产品 → 下一次部署更快更便宜
```

FDE 每次应该沉淀的可复用资产：
- **模板**：Agent 角色定义、Prompt 模板、场景 SOP
- **连接器**：Gmail MCP、Shopify MCP、飞书表 MCP 的配置
- **Eval Case**：这个场景的 Golden Set
- **权限规则**：这个场景的 Policy 规则
- **数据模型**：这个场景涉及的对象和关系
- **工作流组件**：n8n / Temporal workflow 模板

**如果每个客户都做一套独立 Agent → 你在做服务交付。** 每次都能复用更多 → 你在建平台能力。这直接对应原理 3——「判断成败靠第二个 Agent 是否比第一个更快更便宜」。

### 第 3 层：岗位层（最浅但最容易误解的一层）

FDE = 深入现场的工程师 = 懂业务、懂代码、懂交付。

容易被误解成：高级售前 / 解决方案工程师 / 实施顾问 / 客户成功工程师 / 外包开发。

本质区别：这些角色的交付物是「帮客户完成了部署」。FDE 的交付物是「帮客户完成了部署，并且把这次部署的可复用组件反哺给了平台」。

---

## FDE 循环：8 步从现场到平台

```
进入业务现场
→ 找到真实高频痛点（不是听需求表面描述，是看真实工作流）
→ 拆解现有工作流（哪些步骤重复、哪些需要判断、哪些有风险）
→ 判断 AI 介入点（哪些 AI 可以替代、哪些必须人工判断）
→ 找到数据和权限边界（哪些数据可用、哪些权限需要申请、哪些动作不能自动）
→ 做最小可验证原型（能跑通的最小闭环，不是完整系统）
→ 用真实案例测试（拿真实数据和真实场景验证，不是构造的测试数据）
→ 把成功模式沉淀成产品能力（模板、连接器、Eval Case、权限规则）
→ 复制到下一个场景
```

---

## FDE 机制是否有效的 5 个判断指标（按重要性排序）

**第 5 条最关键**：是否降低下一次部署成本？第二个 Agent 是否比第一个更快、更稳、更便宜？

前 4 条衡量的是单次项目的成功：
1. 是否进入真实工作流？（用户每天用，而非演示时用）
2. 是否产生业务增量？（减少人工时间、提高转化、降低错误）
3. 是否有可验证数据？（日志、评估集、人工修改率、失败样例）
4. 是否能沉淀共性组件？（一次项目经验变成可复用的模板）

第 5 条衡量的是能力的累积——这是平台公司和外包公司的本质区别。

---

## 三个常见误解

| 误解 | 正解 |
|------|------|
| FDE 是万能救火工程师 | 如果企业没有基础 IT 系统，FDE 会被拖成需求分析师+后端+数据+客服+项目经理+实施+售后的杂合体。FDE 有效的前提是企业已有基本的信息化基础 |
| FDE 定制越多越好 | 先定制验证价值 → 提炼共性 → 产品化 → 平台化 → 复制。每个客户一套独立系统 = 实施公司，不是产品公司 |
| FDE 能替代 Eval 和治理系统 | FDE 可以发现问题，但不能靠人脑保证系统可靠。仍然需要日志、评估集、权限、审批流、沙箱、回归测试。FDE 是组织机制，治理栈是工程机制，两者互补 |

---

## 一个典型 FDE 场景：邮件回复 Agent

**为什么选这个场景**：
- 业务高频（每天处理大量商务邮件）
- 有历史邮件证据（邮件线程可追溯）
- 风险可控（先只生成草稿，不自动发送）
- 可量化：回复率、草稿可用率（人工修改比例）、日触达人数
- 已有 Gmail OAuth 的基础设施（邮件 Agent 项目）

**FDE 8 步映射到你的场景**：
1. **进入现场** → 你本身就是业务方，你已经在线程里
2. **拆工作流** → 读取邮件→判断合作阶段→判断是否需要报价→生成英文回复→人工审核→发送
3. **判断 AI 介入点** → AI 负责判断合作阶段+生成草稿，人工负责最终发送
4. **数据边界** → Gmail read-only + 草稿模式（不开放发送权限）
5. **最小原型** → 一个能读取历史邮件线程、判断合作阶段、生成回复草稿的 Agent
6. **真实测试** → 拿过去 30 天的历史邮件做 Golden Set
7. **沉淀组件** → 邮件模板库、合作阶段判断规则、风险标记规则 → 下一个 Agent（客服）可以复用
8. **复制** → 客服 Agent 不是在空白上建，而是站在邮件 Agent 的肩膀上

这个场景跑通后，你沉淀的不是「一个邮件 Agent」，而是「Agent 建设方法论 + 可复用组件库」。第二个 Agent（客服）的部署成本应该比第一个低 50-70%。

# 第六部分：技术趋势全景 — 每个机制在完整版图中的位置

## 8 个机制的关系

| 机制 | 职责 | 一句话 |
|------|------|--------|
| **FDE** | 发现真实业务问题 + 把经验反哺系统 | 组织机制 |
| **Context Engineering** | 提供正确、及时、权限合规的上下文 | 数据基础 |
| **MCP** | 让 Agent 调用企业工具 | 工具接口 |
| **A2A** | 让 Agent 之间协作（当前阶段关注即可） | Agent 间通信 |
| **Workflow** | 把 Agent 放进业务流程（不是聊天框，是流程节点） | 执行骨架 |
| **Policy** | 限制 Agent 能做什么 | 安全边界 |
| **Observability** | 记录 Agent 做了什么（Langfuse / OpenTelemetry） | 过程记录 |
| **Eval** | 判断 Agent 做得好不好（Golden Set + 回归评估） | 质量判断 |
| **Human Review** | 高风险决策兜底（报价/退款/合同/正式发送必须人审） | 责任兜底 |

**完整链路**：
```
FDE 发现业务问题 → Context Engineering 准备数据 → MCP 接入工具
→ Workflow 编排流程 → Policy 控制权限 → Agent 执行
→ Observability 记录过程 → Eval 判断质量 → Human Review 兜底高风险
→ FDE 沉淀组件 → 下一次部署更快更便宜
```

## 10 个技术趋势简述

### 1. Context Engineering（从 RAG 升级）
传统 RAG 是「上传文档 → 向量检索 → 回答问题」。企业级 Context Engineering 是「数据来源 → 清洗 → 权限 → 切片 → 元数据 → 检索 → 重排 → 引用 → 时效性 → 冲突处理 → 失效更新」。核心不是谁 Prompt 写得好，而是谁能稳定提供正确、及时、合规的上下文。

### 2. MCP（工具接入标准化）
MCP 会成为 Agent 接企业系统的标准接口层（Gmail MCP、Shopify MCP、GitHub MCP、数据库 MCP），但它本身不是安全治理系统。正确结构是 Agent → MCP Gateway → 权限校验 → 参数校验 → 工具调用 → 日志 → 人工审批。

### 3. A2A（Agent 间通信协议）
MCP = Agent ↔ Tool。A2A = Agent ↔ Agent。未来企业有多个 Agent（客服+销售+广告+财务+供应链），需要标准化通信。当前阶段关注即可，不必重度投入。

### 4. AgentOps（Agent 可观测性）
必须知道：用了哪个 Prompt、检索了哪些资料、哪步失败、花了多少 token、人工改了多少。没有 AgentOps 就没有持续优化。

### 5. EvalOps（从感觉测试到回归评估）
每个 Agent 配 Golden Set。每次改 Prompt/知识库/工具/模型，跑一次完整评估。否则无法判断是真变好还是看起来变好。

### 6. Policy-as-Code
从「Prompt 里写不要越权」升级到「系统层强制不允许越权」。OPA/Rego 规则让权限策略变成可审计的代码。

### 7. Human-in-the-loop
低风险自动、中风险生成草稿、高风险必须人工审批、关键动作必须留痕。这是生产级设计，不是退而求其次。

### 8. Workflow Orchestration
Agent 从聊天式 AI 变成流程里的智能节点：事件触发 → Agent 判断 → 查数据 → 调工具 → 生草稿 → 人工审核 → 执行 → 写日志 → 跟进。

### 9. Coding Agent / Spec-driven Development
代码 Agent 的核心竞争力：理解项目边界、只改该改的、知道如何验证、不会乱重构、能生成可审查 PR。

### 10. Vertical Agent（垂直场景 Agent）
跨境电商客服、达人营销、广告复盘、Shopify 运营、视频拆解。优势：场景窄、数据明确、指标明确、风险边界明确、容易评估、容易沉淀 SOP。

# 第七部分：AI 应用开发的完整 SOP — 10 步从零到复盘

这套 SOP 是前六部分的执行层落地。每一步对应前文中的一个或多个概念。

### Step 1：定义业务场景

先问清楚 6 个问题，输出一页纸的场景定义：
- 这个业务要解决什么问题？
- 现在人工怎么做？（不是「应该怎么做」，是「实际怎么做」）
- 痛点在哪里？（哪些步骤最耗时、最易出错、最需要判断）
- AI 可替代哪些环节？（完全替代 / 辅助）
- 哪些环节必须人工判断？（不可替代）
- 用什么指标判断成功？（量化，见 Step 8）

### Step 2：拆解人工流程

把当前人工流程拆成标准步骤：接收输入 → 判断意图 → 查询资料 → 生成结果 → 人工审核 → 执行动作 → 记录日志 → 异常处理。

### Step 3：定义输入和输出

| 输入 | 输出 |
|------|------|
| 用户问题、邮件正文、视频字幕、产品资料、广告截图、订单号、知识库文档 | 标准回复、SOP 检查清单、邮件草稿、视频拆解报告、风险提醒、下一步动作、数据表记录 |

### Step 4：整理知识库和数据源

资料分 5 层：
- **规则类**：政策、合规、禁用话术 → 不可被模型绕过
- **事实类**：产品参数、价格、物流、订单 → 需要时效性标记
- **案例类**：历史邮件、广告案例、视频案例 → 需要结构化标签（场景/结果/教训）
- **流程类**：SOP、检查清单、操作步骤 → 需要版本号和生效日期
- **模板类**：邮件模板、客服回复、Prompt 模板 → 需要定期审计更新

知识条目模板（统一格式保证可检索性）：
```markdown
# 知识条目
标题：
类型：（规则/事实/案例/流程/模板）
适用场景：
来源：（指向 .raw/ 中的原始文件）
更新时间：
核心内容：
标准回答：（有标准答案的场景）
禁止回答：（明确不能说的话）
转人工条件：（什么情况下不能自动处理）
相关链接：（指向相关 wiki 页面）
```

### Step 5：设计 Agent 角色和边界

```markdown
# Agent 角色定义
你是谁：
你负责什么：
你不能做什么：
遇到不确定信息怎么办：
什么时候必须转人工：
输出格式是什么：
```

**为什么这一步不能省略**：Agent 的最大故障模式不是「答错了」，而是「在不该回答的问题上给出了自信的错误答案」。角色定义中的「你不能做什么」和「什么时候必须转人工」是安全兜底的第一道防线。

### Step 6：设计流程架构

基础结构：用户输入 → 意图识别 → 检索知识库 → 调用工具 → 生成结果 → 风险检查 → 人工审核 → 执行动作 → 写入日志

完整结构（生产级）：Input → Classifier → Retriever → Tool Caller → Reasoning/Drafting → Compliance Check → Human Review → Action → Log → Evaluation → Knowledge Update

### Step 7：设置异常处理

| 异常 | 处理 | 为什么这样处理 |
|------|------|-------------|
| 知识库无结果 | **不要编，转人工** | 编造答案比说「我不知道」危险 100 倍 |
| 检索结果冲突 | 引用来源并提醒人工确认 | 矛盾不能靠模型判断哪条对 |
| 模型回答不确定 | 标记「不确定」，转人工 | 勉强给答案 = 大概率错 |
| 工具调用失败 | 返回失败原因，允许重试 | 区分临时失败和永久失败 |
| 涉及报价/合同 | **只生成草稿，不自动发送** | 财务决策的后果不可逆 |
| 涉及医疗/法律/退款/投诉 | 必须转人工 | 合规红线 |

### Step 8：设置评估指标

通用指标：准确率、召回率、人工修改率、转人工率、响应时间、单次成本、自动处理比例、用户满意度、业务转化率。

按场景的关键指标：
| 场景 | 关键指标 |
|------|---------|
| 客服 Agent | FAQ 命中率、错误回答率、人工转接率、平均响应时间 |
| 邮件 Agent | 分类准确率、草稿可用率、人工修改率 |
| 广告 SOP Agent | 错误设置识别率、检查覆盖率、复盘完成率 |
| 视频拆解 Agent | 字幕完整度、关键帧识别准确率、人工修改时间 |

**没有量化指标，无法判断 AI 是否真有价值。**

### Step 9：上线前测试三阶段

这是 ADPS **G3 渐进承诺**模式的具体落地：

- **阶段 1：只生成草稿，不执行动作。** Agent 所有输出进入审核队列，不触碰任何外部系统。
- **阶段 2：低风险任务半自动执行。** 读邮件、分类、检索知识库自动；生成草稿自动但需人工审核后才发送。
- **阶段 3：高稳定任务自动执行，高风险任务人工审核。** FAQ 回复自动，报价和退款草稿仍需人审。

**信任是挣来的，不是假设的。** 跳过阶段 1 直接全自动，day 11 系统被关停的概率极高。

### Step 10：建立复盘更新机制

每周复盘 7 个问题：
- 答错了什么？
- 为什么答错？（知识库缺失 / 切片不好 / Prompt 不清 / 工具失败 / 规则没写 / 模型不够？）
- 需要补充什么知识？
- 需要优化哪些 Prompt？
- 需要新增什么规则？
- 失败案例是否记录到错误案例库？
- 下周迭代方向？

```

│   └─ 3 套起步组合：知识问答 / 执行型 / 多 Agent 审查

├─ FDE 循环：进入现场 → 拆流程 → 判断介入点 → 找到边界 → 最小原型 → 真实测试 → 沉淀组件 → 复制

└─ Context Engineering 在 ADPS 中的位置：感知行 P1-P4 + 记忆行 M2

技术趋势层（往哪走）
├─ Context Engineering（从 RAG 升级）
├─ MCP + Policy（工具接入 + 权限治理）
├─ AgentOps + EvalOps（可观测 + 可评估）
├─ Human-in-the-loop（分级自动化）
└─ Vertical Agent（垂直场景先产生价值）
```
